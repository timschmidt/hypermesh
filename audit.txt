hypermesh kernel audit
======================

Scope
-----

Audit date: 2026-06-22.

Tracked-file inventory reviewed from `git ls-files`: 820 files. Non-corpus tracked files: 66 files, 175043 counted lines including binary/model assets. Fuzz corpus tracked files: 754 individual seed files across three directories. The audit target is to revise hypermesh into a minimal, full-featured exact high-performance provenance-preserving mesh kernel whose primary top-level type is `ExactMesh`.

Reference input: `references.txt` was present and reviewed, but it is currently untracked. It identifies EMBER, interactive robust mesh booleans, mesh arrangements, Levy 2024 CSG, Barki regularized booleans, Bernstein/Fussell exact booleans, octree BSPs, selective Nef complexes, indirect predicates, Shewchuk, Yap, triangle-triangle predicates, BVH/k-DOPs, planar triangulation, CGAL, Manifold, and Solidean as research inputs.

Workspace state note: before this audit, the repo already had modified source/test files and untracked `references.txt`. This audit does not modify source files and should not stage those pre-existing changes.

Global findings
---------------

1. `ExactMesh` already exists and is the right primary top-level type, but the crate root also exposes workspace, policy, request/result, consumer, and handoff APIs: `ExactBooleanWorkspace`, `ExactBooleanRequest`, `ExactBooleanResult`, `ExactRegularizationPolicy`, `ValidationPolicy`, `ExactMeshConsumerDomain`, and more.
2. The kernel contains valuable exact algorithms and data structures: exact mesh storage, retained bounds, validation facts, face-plane facts, broad-phase scheduling, triangle narrow phase, intersection graph events, coplanar overlap graphs, 2D/3D arrangements, cell complexes, simplification, CDT face-cell triangulation, winding, and volumetric classification evidence.
3. The biggest policy/workspace concentrations are `src/boolean.rs` (16923), `src/reports.rs` (8376), `src/workspace.rs` (1272), and much of the handoff/package/proposal/readiness/artifact family. These should move to csgrs or integration crates.
4. Public API should become mesh-kernel shaped: `ExactMesh`, borrowed mesh views, triangle/edge/face views, exact facts, acceleration structures, low-level graph/arrangement builders, and useful errors. Named booleans, regularization selection, downstream package routing, and product readiness policy belong above this crate.
5. Yap alignment is strong in facts/provenance/replay vocabulary, but it is overexposed as many public reports. Keep the fact/provenance discipline while narrowing the API.
6. The current tests are valuable behavior locks but are concentrated in one large integration test and embedded module tests. Refactor should preserve them while moving policy assertions to csgrs-level tests.

Core mesh and validation
------------------------

- `src/lib.rs` (74): too broad for a kernel root. It should keep `ExactMesh`, `Triangle`, exact errors, facts, borrowed views, and maybe low-level algorithm modules; remove high-level boolean workspace/request/result and consumer-domain exports.
- `src/mesh.rs` (446): keep. `ExactMesh` stores exact vertices, triangles, bounds, validation facts, policy, and construction provenance. Reduce direct public handoff/proposal/report conveniences and add borrowed mesh/face/triangle views.
- `src/facts.rs` (587): keep. Mesh, vertex, edge, face-plane, triangle, and validation facts are the Yap-style cache layer.
- `src/validation.rs` (417): keep validation algorithms and useful topology errors. Public `ValidationPolicy` is partly kernel construction policy and partly product acceptance policy; narrow it or move higher-level variants up.
- `src/bounds.rs` (453): keep. Exact AABB facts and sweep candidate generation are core acceleration structures.
- `src/exact_key.rs` (65): keep internal exact rational point/edge bucketing.
- `src/error.rs` (134): keep diagnostic shape. Split kernel errors from unsupported high-level operation diagnostics.
- `src/scalar.rs` (50): lossy f64 import marker belongs at adapter boundary; keep only if isolated from core mesh constructors.
- `src/view.rs` (278): lossy approximate f64 view is an adapter/export boundary. Move public API to csgrs or feature module.
- `src/audit.rs` (328): useful retained-state audit, but public downstream audit summaries should move up unless used as a compact `ExactMesh` replay fact.

Broad/narrow phase and arrangement internals
--------------------------------------------

- `src/narrow.rs` (248): keep. Exact triangle/plane and triangle/triangle classification with `hyperlimit` predicates is core.
- `src/intersection.rs` (237): keep. It joins exact AABB broad phase with retained face-plane and triangle classification; good acceleration boundary.
- `src/graph.rs` (5186): keep as internal/advanced kernel. It retains intersection events, coplanar edge/vertex facts, split topology plans, and validation. Public access should be through borrowed graph views or explicit low-level builders, not workspace sessions.
- `src/arrangement2d.rs` (2105): keep exact planar segment arrangement and overlay machinery for coplanar faces and loop simplification. Boundary policy should be internal or moved up if it describes output preference.
- `src/arrangement3d.rs` (7108): keep 3D arrangement artifact, face cells, lower-dimensional artifacts, volume regions, and provenance. This is a central exact mesh-kernel data structure.
- `src/cell_complex.rs` (2887): keep labeled/selected cell-complex structures and selection internals. Public named-operation selection policy should move to csgrs.
- `src/simplify.rs` (3186): keep deterministic exact simplification and triangulation of selected cell complexes as kernel algorithms. Public operation/policy APIs should narrow.
- `src/cells.rs` (1861): keep exact CDT face-cell triangulation and validation; aligns with Levy/CDT references and EMBER-style local cells.
- `src/loop_triangulation.rs` (1241): keep shared exact loop triangulation and projection checks; this is low-level geometry support.
- `src/construction.rs` (50): keep exact segment/face-plane construction helper.
- `src/topology.rs` (49): keep small topology helpers.
- `src/witness.rs` (195): keep exact representative witness lattice for volumetric cells.

Boolean, selection, and policy modules
--------------------------------------

- `src/boolean.rs` (16923): main refactor target. It mixes named boolean operations, request/policy/result evaluation, shortcuts, exact arrangement attempts, regularization, materialization, many test helpers, and kernel algorithms. Retain graph/arrangement/cell algorithms and certified shortcut facts; move `ExactBooleanRequest`, named operation API, boundary policy, evaluation/materialization orchestration, and public result types to csgrs.
- `src/reports.rs` (8376): public evidence/report layer for exact boolean staging. Most belongs with csgrs workspace APIs. Keep only compact kernel blockers/facts needed by low-level algorithm replay.
- `src/workspace.rs` (1272): move to csgrs. It is explicitly a reusable boolean session caching graph, arrangements, attempts, evaluations, and materializations for a fixed pair. Hypermesh can provide borrowable cache primitives, but not the workspace-level API.
- `src/regularization.rs` (94): high-level regularization/lower-dimensional/unresolved policy. Move public policy to csgrs; keep internal enums if algorithms need them.
- `src/region.rs` (2382): keep exact split-region and triangulation evidence as kernel internals. Named selection/classification policy should be upper-layer.
- `src/volumetric.rs` (468): keep exact winding classification of triangulated volumetric split cells. Selection policy belongs above.
- `src/volumetric_cells.rs` (1007): keep coplanar volumetric-cell evidence and blockers. This is exactly the useful-error shape needed when a certified materializer is missing.
- `src/winding.rs` (1135): keep exact ray-parity reports for closed meshes; useful core classifier with explicit degeneracy blockers.

Adapters, handoff, and downstream routing
-----------------------------------------

- `src/adapter.rs` (675): external/f64/i64 proposal and readiness adapters. Move public adapter workflow to csgrs; keep exact checked constructors in `ExactMesh`.
- `src/artifact.rs` (1124): shared mesh artifact vocabulary for downstream crates. This is cross-crate integration policy, not minimal kernel.
- `src/handoff.rs` (409): exact solid/surface handoff reports for downstream crates. Move to csgrs/handoff layer.
- `src/package.rs` (920): bundled handoff package and consumer domain routing. Move out.
- `src/proposal.rs` (276): proposal acceptance report. Move public proposal workflow out; keep exact replay validation in kernel.
- `src/readiness.rs` (214): consumer readiness summary. Move out.

Reference-driven notes
----------------------

- EMBER is the best near-term direction for high-performance exact local arrangements, early-outs, and parallel/local decomposition.
- Zhou/Jacobson mesh arrangements and Levy 2024 support the current graph -> arrangement -> cell-complex architecture, including winding vectors, co-refinement, radial sorting, and CDT remeshing.
- Attene indirect predicates suggest reducing eager exact point materialization and comparing constructed objects through provenance where possible.
- Shewchuk, Guigue/Devillers, and Moller validate the current predicate-first triangle staging, but exact construction/fact replay should be the authority.
- Nef complexes and regularized boolean papers are semantic references for csgrs policy, not necessarily direct public hypermesh API.

Audit conclusion
----------------

hypermesh has a strong exact mesh-kernel core, with `ExactMesh` already in place. The refactor should preserve exact facts, bounds, graph events, arrangements, cell complexes, triangulation, simplification, winding, and useful blockers. It should move workspace sessions, named boolean policy, broad reports, downstream handoff packages, proposal/readiness routing, and adapter workflow APIs to csgrs or integration layers.
